# 产品需求文档 (PRD) - 情侣健身打卡小程序 MVP

## 1. 产品概述

### 1.1 产品名称
情侣健身打卡小程序 (Couple Fitness Check-in Mini Program)

### 1.2 产品定位
一款面向年轻情侣的微信小程序，通过快速打卡、数据可视化和互动激励，帮助情侣共同坚持健身，增强情感连接。

### 1.3 核心价值主张
- **3秒快速打卡**：最小化操作流程，降低使用门槛
- **情侣互动**：通过共同目标和数据对比增强情感连接
- **可视化数据**：直观展示健身成果，增强成就感
- **实时反馈**：及时通知和互动，强化使用习惯

### 1.4 版本范围
**MVP (最小可行产品)** - 核心功能验证阶段
- 目标用户：1000-5000 初期用户
- 开发周期：8-12 周
- 发布平台：微信小程序

### 1.5 成功指标
- 日活跃用户 (DAU)：≥ 500
- 周留存率：≥ 60%
- 月留存率：≥ 40%
- 平均打卡完成时间：≤ 3 秒
- 用户满意度：≥ 4.5/5.0

## 2. 用户分析

### 2.1 目标用户群体

#### 主要用户
- **年龄**：18-40 岁
- **性别**：男性和女性均等
- **关系状态**：情侣/夫妻
- **地理位置**：一二线城市为主，包括异地情侣
- **健身基础**：健身新手到中级爱好者

#### 用户特征
- 对健身有基本认知和兴趣
- 希望通过运动增进情感连接
- 对数据和成就系统有兴趣
- 微信活跃用户，习惯使用小程序

### 2.2 用户痛点
1. **缺乏动力**：独自健身容易放弃，需要伴侣陪伴和鼓励
2. **异地困扰**：异地情侣无法一起运动，缺乏共同体验
3. **数据混乱**：健身记录分散，难以追踪进度
4. **互动缺失**：缺乏与伴侣的健身互动方式
5. **操作复杂**：现有健身 App 功能过多，操作繁琐

### 2.3 用户场景

#### 场景 1：日常打卡
- 用户完成一次健身活动
- 打开小程序，快速拍照、选择运动类型和时长
- 3 秒内完成打卡
- 伴侣收到通知，查看打卡记录并点赞/评论

#### 场景 2：周末对比
- 情侣周末一起查看本周数据对比
- 看到双方的打卡次数、总时长、运动类型分布
- 互相鼓励，制定下周目标

#### 场景 3：异地陪伴
- 异地情侣各自在家运动
- 使用"一起练"功能同步开始运动
- 实时看到对方的运动时长，互相发送鼓励语
- 运动结束后查看共同数据总结

#### 场景 4：成就解锁
- 用户连续打卡达到 7 天
- 系统自动解锁"坚持者"徽章
- 收到祝贺通知，伴侣也收到通知
- 在个人中心展示徽章

## 4. 数据模型

### 4.1 核心实体

#### User（用户）
```
- user_id: 唯一标识
- wechat_id: 微信 OpenID
- nickname: 昵称
- avatar: 头像 URL
- gender: 性别
- height: 身高 (cm)
- weight: 体重 (kg)
- created_at: 创建时间
- updated_at: 更新时间
```

**说明**：使用独立的 `couple_user` 表，避免与 RuoYi-Vue 框架的 `sys_user` 表冲突。

#### Partnership（情侣关系）
```
- partnership_id: 唯一标识
- user_id_1: 用户 1
- user_id_2: 用户 2
- status: 状态 (pending/active/dissolved)
- created_at: 创建时间
- dissolved_at: 解除时间
```

#### CheckInRecord（打卡记录）
```
- record_id: 唯一标识
- user_id: 用户 ID
- check_in_date: 打卡日期
- exercise_type: 运动类型
- duration: 运动时长 (分钟)
- calories: 卡路里消耗
- photo_url: 照片 URL
- created_at: 创建时间
```

#### Achievement（成就）
```
- achievement_id: 唯一标识
- user_id: 用户 ID
- badge_type: 徽章类型
- unlocked_at: 解锁时间
```

#### Goal（目标）
```
- goal_id: 唯一标识
- user_id: 用户 ID
- goal_type: 目标类型 (weekly/monthly)
- target_value: 目标值
- created_at: 创建时间
- completed_at: 完成时间
```

#### Interaction（互动）
```
- interaction_id: 唯一标识
- record_id: 打卡记录 ID
- user_id: 操作用户 ID
- type: 类型 (like/comment)
- content: 评论内容 (如果是评论)
- created_at: 创建时间
```

## 5. 业务流程

### 5.1 用户注册与配对流程

```
1. 用户打开小程序
   ↓
2. 请求微信授权
   ↓
3. 授权成功 → 创建用户账号 → 自动登录
   ↓
4. 进入配对页面
   ↓
5. 用户选择：
   a) 生成邀请码 → 分享给伴侣
   b) 输入伴侣邀请码 → 发送配对请求
   ↓
6. 伴侣接受请求 → 配对完成
   ↓
7. 进入情侣主页
```

### 5.2 打卡流程

```
1. 用户点击"打卡"按钮
   ↓
2. 选择拍照或从相册选择
   ↓
3. 选择运动类型
   ↓
4. 输入运动时长
   ↓
5. 系统自动计算卡路里
   ↓
6. 用户确认提交
   ↓
7. 上传到服务器
   ↓
8. 保存成功 → 显示成功提示
   ↓
9. 伴侣收到通知
```

### 5.3 数据同步流程

```
1. 用户操作（打卡、点赞、评论等）
   ↓
2. 本地保存 + 上传到服务器
   ↓
3. 服务器处理并返回结果
   ↓
4. 更新本地缓存
   ↓
5. 推送通知给相关用户
   ↓
6. 用户界面实时更新
```

## 6. 非功能需求

### 6.1 性能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 小程序启动时间 | ≤ 3 秒 | 从点击图标到主页加载完成 |
| 主页加载时间 | ≤ 2 秒 | 从打开小程序到显示主页数据 |
| 打卡提交时间 | ≤ 3 秒 | 从点击提交到显示成功提示 |
| 点击响应时间 | ≤ 1 秒 | 用户点击到界面反应 |
| 并发用户数 | ≥ 1000 | 同时在线用户数 |
| 数据同步延迟 | ≤ 5 秒 | 数据更新到对方看到 |

### 6.2 可靠性需求

- 系统可用性：≥ 99.5%
- 数据备份：每日备份
- 故障恢复时间：≤ 1 小时
- 数据一致性：强一致性

### 6.3 安全需求

- 所有通信使用 HTTPS
- 敏感数据加密存储
- JWT 令牌认证
- 敏感词过滤
- 账号注销时完全删除数据

### 6.4 兼容性需求

- 微信版本：≥ 7.0.0
- iOS：≥ 10.0
- Android：≥ 5.0
- 屏幕尺寸：375px - 812px 宽度

### 6.5 可用性需求

- 界面响应式设计
- 支持离线查看缓存数据
- 清晰的错误提示
- 直观的操作流程

## 7. 技术架构

### 7.1 前端架构

**框架**：WeChat Mini Program (原生)
**语言**：JavaScript (ES6+)
**UI 库**：Vant Weapp v1.10.4+
**状态管理**：getApp() 全局变量 + wx.setStorageSync()
**网络**：wx.request() + 自定义请求包装器

### 7.2 后端架构

**框架**：Spring Boot 2.5.15
**语言**：Java 8+
**ORM**：MyBatis + PageHelper
**认证**：Spring Security 5.7.14 + JWT
**数据库**：MySQL 8.0+
**连接池**：Druid 1.2.27
**API 文档**：Swagger 3.0.0

### 7.3 部署架构

```
WeChat Mini Program (Frontend)
        ↓
    HTTPS
        ↓
Spring Boot API (Backend)
        ↓
MySQL Database
```
